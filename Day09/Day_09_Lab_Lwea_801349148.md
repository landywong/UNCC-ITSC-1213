## Overview

This Lab is a 12-question multiple-choice quiz on Java inheritance, built around the `Dessert` / `Sundae` / `MilkShake` hierarchy you saw in today's class demonstration (`Day_09/Class/Demonstration/`). The questions are organised into three groups, **four questions per Student Learning Outcome**:

* **Questions 1-4 (SLO 1)** -- problem-solving and pseudocode quality. You are given a small design scenario and must pick the best inheritance shape.
* **Questions 5-8 (SLO 2)** -- code correctness, efficiency, and implementation. You read a small Java snippet and predict whether it compiles, what it prints, or which line is wrong.
* **Questions 9-12 (SLO 3)** -- documentation, testing, and best-practice judgement. You are asked to recognise sound style and conventions for inheritance.

Each question has five options labelled **A**, **B**, **C**, **D**, **E**, and exactly one is correct. Pen and paper only -- **do not run any code**.

## The hierarchy under test

For reference, the hierarchy you saw in class is shaped like this:

* `Dessert` (concrete parent) -- `protected` fields `flavor`, `servings`, `cone`; getters; setters; `addServing()`; `describe()`.
* `Sundae extends Dessert` -- `private String topping`; constructor calls `super(flavor, servings, cone)`; overrides `describe()` with `@Override` and starts its body with `super.describe();`.
* `MilkShake extends Dessert` -- `private boolean blended`; adds `blend()`; overrides `describe()` the same way as `Sundae`.

## Quiz

### SLO 1: Problem-solving and pseudocode quality

#### Question 1 (SLO 1)

A team is building three menu classes -- `Dessert`, `Sundae`, and `MilkShake` -- and notices that all three need a `flavor`, a `servings` count, and a `cone` flag. Which design is **best**?

* **A)** Declare those three fields independently in each of the three classes.
* **B)** Declare them in `Sundae`, then have `Dessert` and `MilkShake` `extends Sundae`.
* **C)** Declare them once in `Dessert`, mark them `protected`, and have `Sundae` and `MilkShake` `extends Dessert`.
* **D)** Declare them once in a `Dessert` interface and have all three classes `implements Dessert`.
* **E)** Declare a `static` class `DessertFields` and reach into it from each class.

**Your answer:** _C_

#### Question 2 (SLO 1)

You are writing the parameterised constructor

```java
public Sundae(String flavor, int servings, boolean cone, String topping) {
    // body goes here
}
```

Which body is the **best** choice?

* **A)** `this.flavor = flavor; this.servings = servings; this.cone = cone; this.topping = topping;`
* **B)** `super(flavor, servings, cone); this.topping = topping;`
* **C)** `super(); this.flavor = flavor; this.servings = servings; this.cone = cone; this.topping = topping;`
* **D)** `Dessert d = new Dessert(flavor, servings, cone); this.topping = topping;`
* **E)** `this.topping = topping;` (and let Java set the parent fields for me)

**Your answer:** _B_

#### Question 3 (SLO 1)

In `Dessert`, the fields `flavor`, `servings`, and `cone` are declared as `protected`. Why `protected` rather than `private`?

* **A)** `protected` is faster than `private` at run time.
* **B)** `private` does not compile inside a class that has subclasses.
* **C)** `protected` lets subclasses (`Sundae`, `MilkShake`) read and write the inherited fields directly, while external code is still blocked from touching them.
* **D)** `protected` makes the fields accessible from anywhere in the program.
* **E)** Only `protected` fields are inherited; `private` fields are not.

**Your answer:** _C_

#### Question 4 (SLO 1)

You are asked to add a `Cookie` class. A `Cookie` "is-a" dessert with one extra field, `boolean glutenFree`, and overrides `describe()` to add one more line. Which class declaration is **best**?

* **A)** `public class Cookie extends Sundae { ... }`
* **B)** `public class Cookie extends Dessert { ... }`
* **C)** `public class Cookie implements Dessert { ... }`
* **D)** `public class Cookie extends Dessert, MilkShake { ... }`
* **E)** `public class Dessert extends Cookie { ... }`

**Your answer:** _B_

### SLO 2: Code correctness, efficiency, and implementation

#### Question 5 (SLO 2)

What does this snippet print?

```java
Dessert d = new Dessert("vanilla", 2, true);
d.describe();
```

* **A)** `A vanilla dessert with 2 servings, in a cup.`
* **B)** Compile error: `Dessert` is abstract and cannot be instantiated.
* **C)** Runtime exception: `NullPointerException`.
* **D)** `A vanilla dessert with 2 servings, in a cone.`
* **E)** `Dessert{flavor=vanilla, servings=2, cone=true}`

**Your answer:** _D_

#### Question 6 (SLO 2)

Given the `Sundae` override of `describe()` shown in class

```java
@Override
public void describe() {
    super.describe();
    System.out.println("  Drizzled with " + this.topping + ".");
}
```

how many lines of output does the following snippet produce?

```java
Dessert d = new Sundae("chocolate", 3, false, "hot fudge");
d.describe();
```

* **A)** 0 -- `describe()` returns `void`.
* **B)** 1 -- only the parent's `describe` runs because `d` is typed as `Dessert`.
* **C)** 2 -- the override on `Sundae` calls `super.describe()` and then prints the topping line.
* **D)** Compile error: the override is unreachable through a parent reference.
* **E)** Runtime: `ClassCastException`.

**Your answer:** _C_

#### Question 7 (SLO 2)

What does this snippet print?

```java
Sundae s = new Sundae("chocolate", 3, false, "hot fudge");
s.addServing();
s.addServing();
System.out.println(s.getServings());
```

(Note: `Sundae` does not declare its own `addServing()` or `getServings()`.)

* **A)** `3`
* **B)** `4`
* **C)** Compile error: `Sundae` does not declare `addServing()`.
* **D)** Compile error: `Sundae` does not declare `getServings()`.
* **E)** `5`

**Your answer:** _E_

#### Question 8 (SLO 2)

What happens when this code is compiled and run?

```java
Dessert d = new Sundae("chocolate", 3, false, "hot fudge");
String t = d.getTopping();
```

* **A)** **It does not compile**: the `Dessert` class does not declare `getTopping()`, so the compiler refuses the call through a `Dessert` reference, even though the actual object is a `Sundae`.
* **B)** It compiles and prints `hot fudge`.
* **C)** It compiles but throws a `ClassCastException` at run time.
* **D)** It compiles, but `t` ends up `null`.
* **E)** It compiles but prints an empty string.

**Your answer:** _A_

### SLO 3: Documentation, testing, and best-practice judgement

#### Question 9 (SLO 3)

When should you place `@Override` on a method in a subclass?

* **A)** Never -- `@Override` was deprecated in Java 17.
* **B)** Always, on every method, including ones you write yourself.
* **C)** Only on methods that override a method declared in the parent class (or in an interface). The annotation asks the compiler to confirm the override is real, so a renamed or re-signatured parent fails fast.
* **D)** Only on methods that return `void`.
* **E)** Only on methods that call `super.something()`.

**Your answer:** _C_

#### Question 10 (SLO 3)

Which is the **best** Javadoc for the `Sundae` override of `describe()`?

* **A)** `// describe`
* **B)**
    ```java
    /** Override. */
    ```
* **C)**
    ```java
    /**
     * @param topping the drizzle on top of the sundae.
     */
    ```
* **D)**
    ```java
    /**
     * Print a one-line summary of this sundae.
     *
     * Delegates to {@link Dessert#describe()} via {@code super.describe()}
     * and then adds one line that names the topping.
     */
    @Override
    public void describe() { ... }
    ```
* **E)**
    ```java
    /** @return the printed sundae. */
    ```

**Your answer:** _D_

#### Question 11 (SLO 3)

By Java convention, what should the **first line** of a subclass constructor body be?

* **A)** Either `super(...)` or `this(...)` -- and if you write neither, Java inserts a no-argument `super()` call for you.
* **B)** Always `this.field = field;` for the first inherited field.
* **C)** Always `import java.lang.*;`.
* **D)** Always `@Override`.
* **E)** Nothing -- leave the constructor body empty.

**Your answer:** _A_

#### Question 12 (SLO 3)

A field will be set only at construction time and read by both the parent class **and** its subclasses. No code outside the hierarchy should be allowed to write it. What is the **best** visibility?

* **A)** `public`, so anyone can read or write it.
* **B)** `protected`, with no public setter.
* **C)** `private`, with a `public` setter.
* **D)** Package-private (no modifier), with a `public` setter.
* **E)** `static`, so the field is shared across all instances.

**Your answer:** _B_
