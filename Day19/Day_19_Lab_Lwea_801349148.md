### SLO 1: Problem-solving and pseudocode quality

#### Question 1 (SLO 1)

You have just written a `BankAccount.withdraw(double amount)` method that subtracts `amount` from the balance and throws `InsufficientFundsException` when the requested amount is larger than the current balance. You want to convince yourself the method is correct **before** you ship it. Which set of unit tests is best?

* **A)** One test that withdraws a typical amount from a typical balance.
* **B)** One test for every possible balance from 0 to 1{,}000{,}000.
* **C)** One **happy-path** test (typical input, expected behaviour), one **edge-case** test (withdraw exactly the current balance, the boundary between legal and illegal), and one **error-handling** test (withdraw more than the balance, expect the exception).
* **D)** One test that prints the balance after every withdrawal and asks the grader to eyeball the output.
* **E)** No tests are needed; the method is short enough to read.

**Your answer:** _C_

#### Question 2 (SLO 1)

A teammate hands you a 200-line Java file that crashes with a `NullPointerException` somewhere inside its `process()` method. You can reproduce the failure on demand. What is the **best** first move?

* **A)** Rewrite the entire file from scratch, line by line.
* **B)** Read the full stack trace, note the file and line number near the top, and open that line.
* **C)** Add `try { ... } catch (Exception e) {}` around `process()` so the program no longer crashes.
* **D)** Change the input to something else and see if it stops crashing.
* **E)** Comment out random lines until the crash stops, then call the change a fix.

**Your answer:** _C_

#### Question 3 (SLO 1)

You want to track down where in a 60-line method a calculation first goes wrong. Print debugging is your only tool. Which strategy localises the defect fastest?

* **A)** Print the input at the top of the method and the output at the bottom; then guess.
* **B)** Print every variable on every line.
* **C)** Use a binary search: print the value at the midpoint of the method. If it is correct, the defect is in the second half; if not, it is in the first half. Repeat.
* **D)** Print only the final answer with a label like `>>> answer: `.
* **E)** Do not use prints at all; rewrite the method until the defect goes away.

**Your answer:** _C_

#### Question 4 (SLO 1)

You have just fixed a logic bug that was causing your `Inventory.add` method to overcount the size when an exception was thrown mid-add. You want to make sure the same defect does not silently come back later. What is the **best** thing to do **right now**?

* **A)** Email the team to "be careful with `Inventory.add`".
* **B)** Add a comment above the fix that says `// fixed bug 2026-07-23`.
* **C)** Write a small **regression test** that fails on the *old* version of the code and passes on the *new* one, and check it in alongside the fix.
* **D)** Move on to the next ticket; the bug is fixed.
* **E)** Delete the fix and re-introduce the bug deliberately so the team learns it.

**Your answer:** _C_

### SLO 2: Code correctness, efficiency, and implementation

#### Question 5 (SLO 2)

What happens when this program is compiled and run?

```java
public class Demo {
    public static void main(String[] args) {
        String s = null;
        System.out.println(s.length());
    }
}
```

* **A)** It compiles and prints `0`.
* **B)** It does not compile: `null` cannot be assigned to `String`.
* **C)** It compiles and runs to completion, printing nothing.
* **D)** It compiles, then throws `NullPointerException` at run time, with a stack trace pointing at the call to `s.length()`.
* **E)** It compiles, then throws `IllegalArgumentException` at run time.

**Your answer:** _D_

#### Question 6 (SLO 2)

A unit test reads:

```java
public static void testWithdrawEqualsBalance() {
    BankAccount a = new BankAccount(100.0);     // arrange
    a.withdraw(100.0);                          // act
    if (a.getBalance() == 0.0) {
        System.out.println("++++ PASS testWithdrawEqualsBalance");
    } else {
        System.out.println("---- FAIL testWithdrawEqualsBalance");
    }
}
```

Which of the three arrange-act-assert phases is **missing or under-served** in this test?

* **A)** None: the test follows the arrange-act-assert pattern correctly.
* **B)** Arrange is missing.
* **C)** Act is missing.
* **D)** Assert is missing.
* **E)** All three phases are missing.

**Your answer:** _A_

#### Question 7 (SLO 2)

What does this snippet print?

```java
int[] xs = {10, 20, 30};
System.out.println(xs[3]);
```

* **A)** `30`
* **B)** `0`
* **C)** Compile error: arrays do not support direct indexing.
* **D)** **Runtime exception**: `ArrayIndexOutOfBoundsException` (the legal indices are 0, 1, 2).
* **E)** `null`

**Your answer:** _D_

#### Question 8 (SLO 2)

Consider this method:

```java
public static int sum(int[] xs) {
    int total = 0;
    for (int i = 0; i <= xs.length; i++) {   // bug: <= instead of <
        total += xs[i];
    }
    return total;
}
```

What kind of defect is this, and how does it manifest at run time?

* **A)** A syntax error -- the code does not compile.
* **B)** A logic error -- the code compiles and runs, but the loop reads one past the end of the array, throwing `ArrayIndexOutOfBoundsException`.
* **C)** A concurrency defect.
* **D)** A performance defect: the code is correct but unacceptably slow.
* **E)** No defect at all; the code is correct.

**Your answer:** _B_

### SLO 3: Documentation, testing, and best-practice judgement

#### Question 9 (SLO 3)

By Java convention and Beck's arrange-act-assert pattern (Beck, 2002), what should the **internal shape** of a single unit test be?

* **A)** One line that calls the method under test and nothing else.
* **B)** **Arrange** the inputs and any objects the test needs; **act** by calling the method under test; **assert** by comparing the actual result against the expected one and reporting pass or fail.
* **C)** Arrange, then act 10 times in a loop, then assert once at the end.
* **D)** Use only a single assertion at the very top of the method; nothing below it.
* **E)** Skip the assert; the test is a demonstration, not a verification.

**Your answer:** _B_

#### Question 10 (SLO 3)

A constructor `public Item(String name, double weight)` requires that `name` be non-null and non-empty and that `weight` be non-negative. Which is the **best** style of input validation?

* **A)** Trust the caller; do nothing in the constructor.
* **B)** At the top of the constructor body, check the arguments and `throw new IllegalArgumentException("name must be non-null and non-empty")` (or similar) when a precondition is violated. This is the **fail-early** pattern.
* **C)** Catch the bad input silently and substitute a default like `"unknown"` or `0.0`.
* **D)** Wrap every assignment in a `try`/`catch (Exception e)` and ignore the exception.
* **E)** Validate the inputs only when the program runs in a debugger.

**Your answer:** _B_

#### Question 11 (SLO 3)

Print debugging done well in an introductory Java program looks like which of the following?

* **A)** Add `System.out.println(x);` lines and leave them in the code forever.
* **B)** Add a single print at the very top of `main` that says `"hello"`.
* **C)** Add prefixed, contextual prints (for example `">>> top of inner loop: i=" + i + ", j=" + j + ", arr=" + Arrays.toString(arr)`), use them to localise the defect, and **remove them once the defect is fixed**.
* **D)** Print only inside `catch` blocks.
* **E)** Print only the final answer; everything else is noise.

**Your answer:** _C_

#### Question 12 (SLO 3)

Which of the following is the **best** working definition of a regression test, in the sense used by Zeller (2009) and by the testing literature in general?

* **A)** A test that runs slower over time and warns when the program has regressed in performance.
* **B)** A test that is written **after** a defect has been fixed, fails on the pre-fix code and passes on the post-fix code, and is checked in alongside the fix so the same defect cannot silently come back later.
* **C)** A test that the build system runs only on Fridays.
* **D)** A test that is automatically generated from the production code.
* **E)** A test that re-runs every previous test in reverse order.

**Your answer:** _B_
