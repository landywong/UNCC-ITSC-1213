## Quiz

### Inheritance and method overriding

#### Question 1

Open `Player.java`. How does the class declaration begin?

* **A)** `public class Player`
* **B)** `public class Player implements Entity`
* **C)** `public class Player extends Combatant`
* **D)** `public class Player extends Entity implements Combatant`
* **E)** `public abstract class Player extends Entity`

**Your answer:** _D_

#### Question 2

The parameterised constructor of `Player` begins with the line `super(name, health);`. What is the role of that line?

* **A)** It calls `Player`'s own no-argument constructor recursively.
* **B)** It calls the parent's parameterised constructor `Entity(String, int)` to initialise the inherited fields `name` and `health`. Java requires this call to appear as the **first** statement of any subclass constructor that needs to invoke a parent constructor with arguments.
* **C)** It declares two new local variables called `name` and `health`.
* **D)** It is optional and the program would behave identically without it.
* **E)** It throws an `IllegalArgumentException` if either argument is negative.

**Your answer:** _B_

#### Question 3

`Player`'s `display()` method carries the `@Override` annotation. What is the **best** reason to write `@Override` on that method?

* **A)** It marks the method as abstract.
* **B)** It makes the method run faster at runtime.
* **C)** It tells the compiler to confirm that the method really does override one declared in the parent (or in an interface). If the method's name or signature did not match anything to override, the compiler would refuse to compile the class -- catching the typo at compile time rather than as a silent runtime bug.
* **D)** It marks the method as `static`, which lets the JVM call it without an object.
* **E)** It is a comment that the compiler ignores.

**Your answer:** _C_

### Abstract classes

#### Question 4

Open `Entity.java`. How does the class declaration begin?

* **A)** `public class Entity`
* **B)** `public abstract class Entity`
* **C)** `public interface Entity`
* **D)** `public final class Entity`
* **E)** `public static class Entity`

**Your answer:** _B_

#### Question 5

Suppose a student adds the line `Entity e = new Entity("Mystery", 50);` to `main`. What happens?

* **A)** The line compiles and runs; `e` is a new `Entity` object with `name = "Mystery"` and `health = 50`.
* **B)** The line compiles, but throws `IllegalArgumentException` at runtime.
* **C)** The line compiles, but throws `NullPointerException` at runtime.
* **D)** The line **does not compile**, because `Entity` is an abstract class and abstract classes cannot be instantiated with `new`.
* **E)** The line compiles but produces no observable effect.

**Your answer:** _D_

#### Question 6

Inside `Entity.java`, the method `display()` is declared `public abstract void display();` (no body, just a semicolon). What does that declaration require of `Player` and `Monster`?

* **A)** They must each provide a concrete body for `display()` (or be declared `abstract` themselves). The compiler enforces this -- a concrete subclass that does not implement an inherited abstract method will not compile.
* **B)** They must mark `display()` as `static`.
* **C)** They may inherit `display()` unchanged with no further code; the parent's empty body is what runs.
* **D)** They may implement `display()` only if `Entity` is also marked `final`.
* **E)** They are forbidden from overriding `display()`.

**Your answer:** _A_

### Interfaces

#### Question 7

Open `Combatant.java`. How does the file's top-level declaration begin?

* **A)** `public class Combatant`
* **B)** `public abstract class Combatant`
* **C)** `public interface Combatant`
* **D)** `public enum Combatant`
* **E)** `public record Combatant`

**Your answer:** _C_

#### Question 8

How many methods does `Combatant` declare, and what is the body of each?

* **A)** Two methods, `attack` and `defend`, each with a full body.
* **B)** One method, `attack(Entity target)`, declared as a header terminated with a semicolon and with no body. Methods declared inside an interface are implicitly `public` and `abstract`; any class that implements the interface must supply a body.
* **C)** One method, `attack`, with a default implementation that prints `"attacking"`.
* **D)** Zero methods; `Combatant` only declares constants.
* **E)** Three methods: `attack`, `defend`, and `flee`.

**Your answer:** _B_

#### Question 9

How does each of `Player` and `Monster` relate to `Combatant`?

* **A)** Each class extends `Combatant` with `extends Combatant` in its header, and inherits the implementation of `attack` from the interface.
* **B)** Each class declares a field of type `Combatant` and delegates `attack` calls to it (composition).
* **C)** Each class declares `implements Combatant` after its `extends Entity` clause and provides its own `attack(Entity)` method, annotated with `@Override`. The two `attack` implementations differ: `Player.attack` deals damage proportional to the player's level, while `Monster.attack` deals the monster's fixed `damage` value.
* **D)** Neither class is related to `Combatant`; the interface is unused.
* **E)** Each class redeclares the entire `Combatant` interface inside its own file.

**Your answer:** _C_

### Polymorphism

#### Question 10

In `Main.main`, the line

```java
Entity[] entities = { player, monster };
```

stores a `Player` and a `Monster` in an array typed as `Entity[]`. What property of the type system makes this assignment legal?

* **A)** Java performs an explicit cast on each element automatically and silently.
* **B)** Both `Player` and `Monster` *is-a* `Entity` (via inheritance: `Player extends Entity` and `Monster extends Entity`). A subclass reference is implicitly upcast to a parent-typed slot, so a `Player` and a `Monster` both fit in an `Entity[]` slot without any cast.
* **C)** The array's runtime type is silently changed to `Object[]` so that anything can be stored in it.
* **D)** It is a special-case rule that only applies to arrays of length 2.
* **E)** It is illegal; the line does not compile.

**Your answer:** _B_

#### Question 11

The for-each loop at the top of `main` runs `e.display();` on each element of `entities`. The loop variable `e` is statically typed `Entity`. Which `display()` method actually executes for each element?

* **A)** `Entity.display()` runs for every element, because `e` is typed as `Entity` -- it would print the same generic message twice.
* **B)** A compile error: `display()` is abstract in `Entity`, so the call is not allowed.
* **C)** Whichever `display()` is overridden on the **object's actual class** runs each iteration: `Player.display()` for the player and `Monster.display()` for the monster. This is dynamic dispatch -- the runtime type of the object, not the static type of the variable, decides which override is executed.
* **D)** `Player.display()` runs for both elements because `Player` is the first slot in the array.
* **E)** Nothing runs, because abstract methods are skipped at runtime.

**Your answer:** _C_

#### Question 12

Suppose we add the line

```java
String ability = entities[0].getSpecialAbility();
```

immediately after the array is built. What happens, and what is the **most accurate** description of what would have to change for the program to print the player's special ability through an `Entity` reference?

* **A)** The line compiles and prints `"claw"` because `Entity` declares `getSpecialAbility`.
* **B)** The line throws `NullPointerException` at runtime because `entities[0]` is null.
* **C)** The line **does not compile**, because `Entity` does not declare `getSpecialAbility`. To reach a `Player`-only method through a parent-typed reference, the program must guard a downcast with `instanceof`: `if (entities[0] instanceof Player) { ((Player) entities[0]).getSpecialAbility(); }`.
* **D)** The line compiles but always returns `null`, regardless of which subclass is in slot 0.
* **E)** The line compiles only if `Entity` is also marked `Combatant`.

**Your answer:** _C_
